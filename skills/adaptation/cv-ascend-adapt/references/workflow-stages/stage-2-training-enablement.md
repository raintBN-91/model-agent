# 阶段2：训练链路打通

## 输入信息

- 环境收敛结果
- 数据集定位结果
- 依赖安装与验证结果
- 训练入口脚本、配置文件、runner 构建路径
- 用户对单卡、多卡、AMP、收敛验证的要求

## 必做动作

- 优先审查训练入口脚本，而不是先改模型实现
- 修改训练入口脚本时，必须显式补充以下 NPU 运行时导入：

```python
import torch_npu
from torch_npu.contrib import transfer_to_npu
```

- 在训练入口补充 NPU 可用性检查与设备绑定
- 检查是否在 runner 构建前执行 `torch.npu.is_available()`
- 检查是否在 runner 构建前执行 `torch.npu.set_device(...)`
- 检查是否把实际设备信息写回配置对象，供 runner 和 hook 使用
- 统一参数：
  - `--use-npu`
  - `--device`
  - `--num-devices`
  - `--master-port`
  - `--use-amp`
- 启动验证分三层进行：
  - 单卡启动校验
  - 真实图片 1 epoch 验证
  - 长周期收敛验证（按用户要求执行，例如 50 epoch 或 100 epoch）
- 若用户要求真实验证，至少继续完成：
  - 单卡真实图片 1 epoch 训练
  - 至少一次真实验证流程
  - 必要时执行长周期训练以判断 loss 是否收敛
- 若当前环境不能执行真实 NPU 命令，则只完成脚本生成与执行说明，由客户复制到昇腾机器执行

## 禁止动作

- 训练链路未审查入口脚本就直接改模型实现
- 未补充 `torch_npu` 导入和运行时检查就推进训练
- 在 `torch.npu.is_available()` 检查前构建 runner
- 在设备绑定前启动单卡或多卡训练
- 只验证“能启动”就判定训练链路打通
- 在当前环境不能执行真实 NPU 时伪造真实训练结论

## 完成标准

- 单卡启动任务通过
- 训练任务不仅能启动，而且能迭代
- 至少一次真实验证流程可完成，或已明确转为客户侧执行交付
- 若用户要求收敛验证，已执行长周期训练并具备 loss 收敛证据

## 失败处理

- 若单卡启动失败，先修复入口脚本、设备绑定、参数传递问题
- 若能启动但不能迭代，继续排查数据、dtype、设备写回、hook 依赖问题
- 若当前环境不可真实执行，停止真实训练并转为脚本与执行说明交付
