# NPU 适配决策树与工程经验

## 适配新模型决策流程

```
拿到一个新模型
│
├─ 它是纯 PyTorch 模型吗？
│   └─ 是 → 类型 A：.to("npu:0") 即可，重点解决权重下载
│
├─ 它是框架封装的模型吗？
│   └─ 是 → 类型 B：找到框架的 device 参数，传入 "npu:0"
│
├─ 它依赖 CUDA kernel 吗？
│   └─ 是 → 类型 C：找 OpenCV/PyTorch 等价实现，不跑原模型
│
├─ 它在 NPU 上报错但不该报错？
│   └─ 是 → 类型 D：定位算子 → 找数学等价实现 → monkey-patch
│
└─ 不确定？
    └─ 先类型 A 尝试，失败后看报错信息判断属于哪类
```

---

## 优雅降级策略（三层回退）

所有模型都遵循：

```
第一层：尝试加载预训练权重
   ↓ 失败（网络不通/权重不存在）
第二层：from_config 随机初始化
   ↓ 失败（config 也有问题）
第三层：回退到功能等价的替代实现
   ↓ 失败
标记为不可用，记录原因
```

**关键认知**：没有预训练权重，模型输出是随机的，但这**不影响验证 NPU pipeline 的正确性**。shape、dtype、算子兼容性都不依赖权重值。

## 输入 Shape 排查指南

输入 shape 是最高频的报错原因。

```python
# 1. 先查文档/源码，确认输入格式
# 2. 用最小的合法尺寸构造输入
# 3. 如果报错，检查：
#    - 维度数量对不对（3D/4D/5D）
#    - 通道数对不对
#    - 图片模型：(B, C, H, W) 还是 (B, H, W, C)
#    - 视频模型：(B, C, T, H, W) 还是 (B, T, C, H, W)
#    - 音频模型：(B, T) 还是 (B, C, T) 还是 (B, T, F)
#    - dtype 对不对（float32 / long / uint8）
```

## 数据策略：合成 vs 真实

| 场景 | 策略 |
|------|------|
| 首次适配，不知道模型能否加载 | 合成数据（torch.randn），先跑通再说 |
| 验证 shape 和算子兼容性 | 合成数据，够用 |
| 验证预处理管线 | 必须真实数据（图片/音频文件） |
| 验证输出质量/精度 | 必须真实数据 + 预训练权重 |
| 给他人展示效果 | 真实数据 + 可视化 |

---

## 适配成功三层次判定

| 层次 | 标准 | 适用场景 |
|------|------|---------|
| **冒烟通过** | 合成数据 forward 不报错 | 快速验证 NPU 兼容性 |
| **pipeline 通过** | 真实数据 forward 不报错 | 验证预处理+推理管线 |
| **可用** | 真实数据 + 预训练权重 + 输出合理 | 生产环境评估 |

大多数模型只需"冒烟通过"或"pipeline 通过"。

## 目录结构自包含原则

```
<模型名>/
├── test_<模型名>.py      # 必选：冒烟测试
├── infer_<模型名>.py     # 可选：真实推理
├── <模型名>_result.json  # 冒烟测试输出
├── infer_result.json     # 真实推理输出
├── viz.png               # 可视化输出
├── README.md             # 说明文档（含适配坑点）
├── test.jpg / test_audio.wav  # 测试素材
```

**原则**：复制这个目录到另一台 NPU 机器上，`python3 test_xxx.py` 就能跑。
