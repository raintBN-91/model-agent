# 常见误区

## 目的

本文件用于避免精度对齐中最容易出现的流程性错误，尤其是基线选择错误、CPU 对比越权和报告范围夸大。

## 误区一：未确认官方基线就先写比较脚本

错误做法：

- 先写 `CPU vs NPU` 或其他本地比较脚本
- 跑完后再回头看 README 有没有基线

正确做法：

- 先查官方 README、官方示例、benchmark 或官方文档
- 确认官方是否已经给出标准结果
- 只有官方基线不足时，才向用户确认 GPU 基线

## 误区二：因为 CPU 更容易跑通，就默认做 CPU vs NPU

错误做法：

- 本地有 PyTorch 模型，就直接跑 CPU 结果当基线
- GPU 不方便获取，就默认用 CPU 顶上

为什么不对：

- CPU 路径并不自动等于官方基线
- CPU 对比最多只能帮助定位 PyTorch/NPU 路径问题
- 它不能证明 NPU 与官方模型、原始框架或用户 GPU 结果一致

正确做法：

- 先官方基线
- 再用户 GPU 基线
- CPU 只在用户明确要求或仅作调试时使用

## 误区三：先给 PASS，再回头修流程

错误做法：

- 先基于错误基线给出 PASS
- 用户指出流程不对后，再去补官方基线

正确做法：

- 在基线来源未确认前，不要给正式对齐结论
- 如果前面做了调试性比较，也只能写成“临时调试结果”

## 误区四：看到 mismatch 后临时放宽阈值

错误做法：

- 先按一个阈值比较，发现 mismatch
- 然后把阈值放宽，再直接写“全部 match”

正确做法：

- 阈值应在比较前声明
- 如果确实需要调整阈值，必须说明依据
- 报告中要保留原始 mismatch 事实，不能直接抹掉

## 误区五：样例对齐就宣称整个模型已完成精度对齐

错误做法：

- README 里只有少量推理样例
- 样例比对通过后，就写成“模型完成精度对齐”

正确做法：

- 官方样例只证明样例覆盖范围内的推理结果基本一致
- 训练对齐、完整评测集对齐、用户 GPU 基线对齐都需要单独说明

## 推荐表述

当证据只覆盖官方样例时，优先写成：

- “官方 README 样例级推理结果基本对齐”
- “当前尚不能据此宣称整个模型已完成训练和推理层面的全面精度对齐”

当只做了 CPU 调试对比时，优先写成：

- “当前仅完成 CPU vs NPU 的调试性比较”
- “该结果不能替代官方基线或用户 GPU 基线”
