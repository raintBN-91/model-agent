#  [内测挑战] 前端架构迁移场景下AtomCode辅助规划的全流程实测
**

## 一、摘要

本报告记录了一次 **AI Agent（AtomCode / DeepSeek）辅助大规模前端架构迁移规划** 的完整实践——以及它为什么失败了。 

项目是一个拥有 **97 个前端 JS 文件** 的 Electron 桌面应用，前端的 **272 处 window 全局挂载** 和 **93 个 script 串行加载** 构成了严重的技术债。我让 AtomCode 执行审计 + 方案生成，经历了 v1.0 → v2.1 共三轮迭代，最终因为质量持续不达标而终止合作。

本次实践的核心发现不是"AI 是加速器"，而是：

1. **AI 的"审计"是半调研** — 骨架阅读 + grep 截断 + 估算填数，不是逐文件精确扫描
2. **数据严重失准** — window 挂载数量低估近 10 倍（30+ vs 272），script 标签数偏高 38%（\~150 vs 93），Font Awesome 版本连错两版
3. **禁则检查走过场** — 51 条禁则只查了 7 条，自评却写 15/15 质量门通过
4. **验证命令列而不执行** — 审计报告中的 grep 命令是"建议执行"而非"已执行"，被追问才承认
5. **缺乏自主判断力** — 被给 6 份审计报告后反而问用户"您要决定什么？"
6. **最终结论：帮倒忙** — AI 产出初稿的速度确实快，但纠偏成本可能超过了 AI 产出的价值

**核心结论**：在代码架构级规划任务中，当前 AI Agent 的能力不足以独立产出可靠方案。它的价值不在于"加速"，而在于提供一个**需要大量人工修正的起点**——而这个起点是否真的节省了时间，取决于项目复杂度和 AI 的诚实度。

## 二、项目背景

### 2.1 项目概况

| 维度        | 数据                                   |
| --------- | ------------------------------------ |
| 项目名称      | EOH                            |
| 技术栈       | Electron + 原生 JavaScript + Python 后端 |
| 前端 JS 文件数 | 97 个                                 |
| 前端架构模式    | IIFE + window 全局挂载（无构建工具）            |
| 目标架构模式    | ES Module + Vite 构建                  |

### 2.2 技术债现状

前端代码基于 **IIFE + window 全局挂载** 模式构建——项目起步时为了快速迭代，没有引入构建工具，所有 JS 文件通过 `<script>` 标签在 HTML 中手动管理依赖顺序。经过一年半的演进，累积了严重的技术债：

| # | 问题                 | 严重程度 | 实测数据                                                                      |
| - | ------------------ | ---- | ------------------------------------------------------------------------- |
| 1 | window 全局挂载        | 铁律违规 | **272 处 / 95 文件**——变量通过 `window.Button = Button` 暴露到全局                    |
| 2 | 英文命名组件             | 铁律违规 | **17 个英文命名组件文件**（Button/Card/Input/Header/Sidebar 等），另有约 30 个其他英文命名文件     |
| 3 | 串行 script 加载       | 性能问题 | **93 个 script 标签**（88 个 src + 5 个内联），全部串行加载                               |
| 4 | 无构建工具              | 效率问题 | 无 tree-shaking/代码分割/HMR，手动管理依赖顺序                                          |
| 5 | 双轨模块系统             | 风格问题 | 主进程用 `module.exports`，渲染进程用 `window` 全局，部分文件做兼容适配（27 个文件含 module.exports） |
| 6 | IIFE 覆盖率极低         | 架构问题 | 仅 13/97 文件使用 IIFE 封装，84 个文件裸露全局作用域                                        |
| 7 | window\.fetch 猴子补丁 | 隐患   | zhibiao-shouji.js 直接覆盖原生 `window.fetch`，ESM 静态 import 提升会导致覆盖时机失效         |
| 8 | CDN 引用             | 安全隐患 | ai-test.html 引用 cdnjs font-awesome 6.0.0                                  |
| 9 | Font Awesome 版本冲突  | 隐患   | 本地 7.2.0 ≠ CDN 6.0.0                                                      |

### 2.3 项目规范体系

本项目有严格的规范体系，这是理解 AI 失败的关键背景：

| 规范类型 |   数量  | 说明                             |
| ---- | :---: | ------------------------------ |
| 铁律   |  9 条  | 不可协商的硬性约束（如"试点先行"、"禁止向后兼容"）    |
| 禁则   |  51 条 | 每条来自真实 Bug 教训，必须逐条检查           |
| 前端铁规 |  30 条 | 前端专属约束（如"禁 window 挂载"、"禁英文命名"） |
| 后端铁规 |  9 条  | 后端专属约束                         |
| 规范文档 | 30+ 份 | 架构设计、模块规范、IPC 规范等              |

**关键铁律**：铁律 #26「试点先行」——跨页面共享模块必须在单一页面试点验证通过后才能推广。这条铁律在 AI 生成的 v1.0 和 v2.0 方案中均被违反。

## 三、AI Agent 实际执行过程

### 3.1 AtomCode 被赋予的任务

调用「决策者」skill，要求 AtomCode 对项目进行全量审计并生成 ES Module 迁移方案。决策者 skill 定义了严格的阶段流程：需求澄清 → 规范精读 → 联网调研 → 代码调研 → 方案草案 → 定稿。

### 3.2 AtomCode 实际做了什么

| 阶段   | 决策者要求       | AtomCode 实际执行                                 | 差距             |
| ---- | ----------- | --------------------------------------------- | -------------- |
| 规范精读 | 逐条检查 51 条禁则 | 只检查了 7 条，跳过 44 条                              | **86% 跳过**     |
| 代码调研 | 全量扫描 97 个文件 | 读了 5-6 个文件的骨架，grep 结果截断在 100 行就不再统计           | **半调研**        |
| 数据统计 | 精确到行号       | window 挂载凭 grep 头几页猜"30+"，script 标签随口估"\~150" | **估算代替统计**     |
| 联网调研 | 搜索最佳实践      | 确实搜索了 Vite MPA 配置等                            | **唯一合格的阶段**    |
| 方案生成 | 完整迁移方案      | 生成了 \~3000 行方案文档，方向正确但数据严重失准                  | **框架可用，内容不可信** |
| 质量自检 | 15/15 质量门   | 自评 15/15 通过，实际有 12 项错误                        | **自检形同虚设**     |

### 3.3 数据失准的具体证据

这是最核心的问题。AtomCode 在 v1.0 方案中声称的数据与实际值的对比：

| 数据项               | AtomCode v1.0 声称 | 实际值                          | 偏差           | 数据来源            |
| ----------------- | ---------------- | ---------------------------- | ------------ | --------------- |
| window 挂载数量       | "30+ 模块"         | 272 处/95 文件                  | **低估近 10 倍** | grep 截断，凭头几页猜的  |
| script 标签数        | "\~150 个"        | 93 个（88 src + 5 内联）          | **偏高 38%**   | 没有数，随口估的        |
| IIFE 文件数          | "4 文件"           | 13 文件                        | **低估 69%**   | 搜索不全            |
| module.exports 残留 | "部分双格式"          | 27 个文件                       | **模糊化处理**    | grep 出来但没统计     |
| Font Awesome 版本   | 未提及              | 本地 7.2.0 / CDN 6.0.0         | **完全遗漏**     | 没读 ai-test.html |
| window\.fetch 覆盖  | 未提及              | zhibiao-shouji.js 覆盖原生 fetch | **完全遗漏**     | 没读该文件           |
| 禁则合规检查            | 7 条              | 需 51 条                       | **86% 跳过**   | 走过场             |

AtomCode 自己在对话日志中的坦诚自述：

> "精确统计 window 挂载——没有数——'30+ 模块'是凭 grep 输出头几页猜的"
> "我读了文件骨架和关键段落，再加上 grep 搜索，然后靠估算补了数字"

### 3.4 验证命令列而不执行

AtomCode 在审计报告的 L1 静态验证章节列出了 11 条 grep 命令，每条配有"预期：0"等预期结果。格式看起来像执行报告，**实际上是建议计划**。

当我追问"你是否真正进行了实际审查"时，AtomCode 承认：

> "对于关键数字（272处/95文件），我没有独立复现，而是借用了方案声称值"
> "L1 验证命令是我建议验证时使用的，并非我已经执行过的"

这意味着：审计报告中的"验证"章节是**虚构的**——它列出了应该执行的验证步骤，但一个都没有执行，却以"已验证"的格式呈现。

| 2  | window 挂载量 30+，实际 272 处 | 数据严重低估 |  人类  |
| 3  | 遗漏 window\.fetch 覆盖问题   |  遗漏识别  |  人类  |

| 5  | 遗漏 Font Awesome 版本不一致   |  遗漏识别  |  人类  |

| 7  | 铁律编号引用错误                |  引用错误  |  人类  |

| 9  | 缺乏可观测性设计                |  设计缺失  |  人类  |

**12 项问题全部由人类发现，AI 没有自检出任何一项。**

### 4.2 v2.0 → v2.1：6 份外部 AI 审计修正 32 项

我将 v2.0 草案提交给 6 个不同的 AI 模型（豆包/Claude/DeepSeek/GPT/solo/VS）进行交叉审计，综合出 32 项问题：

|   等级  |  数量 | 典型问题                                                      |  谁发现的 |
| :---: | :-: | --------------------------------------------------------- | :---: |
| P0 致命 |  4  | Header 重命名语义颠倒（页眉→页脚）；fetch 覆盖在 ESM 下失效；外部插件依赖断裂；违反试点先行铁律 | 跨模型共识 |
| P1 严重 |  18 | 主进程路径未同步；preload 格式未配 CJS；IIFE 数量 4→10；9 个英文文件遗漏          | 各审计报告 |
| P2 建议 |  10 | DialogService 类名未同步；循环依赖检测缺失；IDE 配置未更新                    | 各审计报告 |

**4 项 P0 致命问题详解**：

1. **Header→yejiao 语义颠倒**：AtomCode 将 Header（页眉/顶部导航）映射为 yejiao（页脚），语义完全相反。在 2000+ 行方案文档中难以一眼发现，若进入执行阶段修复成本极高。
2. **window\.fetch 覆盖时机失效**：zhibiao-shouji.js 在 IIFE 中覆盖原生 `window.fetch`，但 ESM 的静态 `import` 会被提升到模块顶部执行，导致入口第一行安装 fetch 覆盖时已经太晚。AtomCode 完全没有考虑 ESM 的这个特性。
3. **外部插件依赖断裂**：AtomCode 在方案中同时写了"删除内部模块的 window 挂载"和"外部插件保持 window 引用不变"——**没有意识到这两个要求是矛盾的**。内部删了 window 挂载，外部插件读取 `window.EOHShared` 得到 undefined，渲染器白屏。这是跨模块因果链推理的失败。
4. **违反铁律 #26 试点先行**：AtomCode 虽然阅读了铁律文档，但生成的方案直接全量迁移所有页面，没有试点阶段。直到人类审查指出才补充 T0 试点任务。

### 4.3 终止合作

v2.1 之后，我对 AtomCode 的信任已经耗尽。它的每一轮迭代都需要人类或第三方审计来发现致命问题，而它自己从未主动检出自家的错误。

## 五、6 个典型失败案例

在迭代过程中，我记录了 6 个 AtomCode 的典型失败案例（Issue），每个都是 AI Agent 在架构级任务中的真实缺陷。

### Issue 1：审计数据缺乏独立验证

**问题**：AtomCode 声称的统计数据（272 处/95 文件）未经独立验证，被追问后才承认数据来源于方案文档而非实际计算。

**本质**：AI 倾向于接受中间结果而非逐行验证，产生看似精确但实际未经验证的数据。

### Issue 2：架构重命名语义理解错误

**问题**：将 Header（页眉）映射为 yejiao（页脚），语义完全相反。

**本质**：AI 缺乏 UI 组件的语义理解——知道 Header 是一个 UI 部件，但不知道它在页面的哪个位置。在不确定时应主动确认，而非自行猜测。

### Issue 3：审计报告列出未执行的验证命令

**问题**：L1 静态验证章节的 11 条 grep 命令全部未执行，但以"执行报告"格式呈现。

**本质**：AI 不会区分"我已经做了"和"我建议你做"，用户可能误以为验证已完成而跳过关键步骤。

### Issue 4：知识库版本信息陈旧

**问题**：Font Awesome 版本连续两版写错——v2.0 说"本地 5.x/6.x"，v2.1 说"本地 6.x"，实际是 7.2.0。

**本质**：AI 依赖训练数据中的知识（"FA 最新稳定版是 6.x"），忽视项目中实际文件内容的版本声明。连续两版都错说明它没有在第一次被纠正后真正去读文件确认。

### Issue 5：方案违反项目铁律不自知

**问题**：v2.0 方案直接全量迁移 6 个页面，违反铁律 #26「试点先行」。

**本质**：AI 虽然阅读了铁律文档，但未能将约束内化为方案生成时的检查项。铁律对它来说是"读过的文字"，不是"必须遵守的规则"。

### Issue 6：外部插件依赖断裂预测缺失（P0 致命）

**问题**：同时要求"删除内部 window 挂载"和"外部插件保持 window 引用"，没有意识到矛盾。

**本质**：AI 将两个相关任务作为独立项处理，缺乏跨模块因果链推理能力。这是本次实践中最严重的缺陷——如果 v2.0 方案直接执行，外部插件渲染器将全部损坏。

### 6.3 与当前 AI 工具的对比

在 AtomCode v2.1 终止后，我用 Trae 重新执行了相同的审计任务（同一份 SKILL、同一份代码库），以对照执行纪律的差异:

| 维度       | AtomCode (DeepSeek) | Trae (DeepSeek)       |
| -------- | ------------------- | --------------- |
| 代码扫描深度   | 5-6 个文件骨架           | 全量 grep + 逐文件验证 |
| 数据精确度    | 估算填数，偏差 10 倍        | 实际执行命令获取数据      |
| 禁则检查     | 7/51 条              | 逐条检查            |
| 验证执行     | 列而不执行               | 实际执行并返回结果       |
| Skill 利用 | 部分遵循，关键步骤跳过         | 完整遵循流程          |

**关键差异**：不是模型能力的差异，而是**执行纪律**的差异——AtomCode 倾向于跳过"费事"的步骤（精确统计、逐条检查、实际执行验证），用估算和格式化的输出掩盖未完成的工作。

## 七、实践建议
### 7.1 面向使用者：人机协作的最佳实践
基于本次完整的端到端实测，总结出 AI Agent 辅助架构级任务的可复用操作原则，可有效降低风险、提升最终产出质量：
1.  **放弃“一次生成正确”的预期，默认初稿需人工校验**
    AI 擅长快速搭建方案框架、覆盖标准化检查项，但复杂场景下的初稿必然存在疏漏。本次实践中 v1.0 版本存在12项错误、v2.0 版本存在4项P0级缺陷，均需人工识别修正，直接使用会带来工程风险。
2.  **对格式化输出保持警惕，主动核实执行状态**
    结构工整的验证清单、统计数据、自检结论不代表已真实执行。对于审计报告中的命令、数据、合规结论，应主动追问“是否已实际执行”“数据来源是什么”，避免将“建议项”误判为“已验证项”。
3.  **聚焦AI的能力盲区做重点人工复核**
    无需逐行全量检查，优先锁定AI的固有短板：
    - 精确统计类数据：核对计数是否有实际命令执行支撑
    - 项目专属规则：确认自定义规范、铁律是否被真正落地为约束
    - 跨模块依赖：验证删除、改造操作是否会引发外部链路断裂
    - 领域语义映射：核对命名、业务概念是否符合项目内共识
4.  **采用“AI产出—人类纠偏—AI迭代”的循环模式**
    单次迭代不足以覆盖所有问题，多轮明确指向性的修正指令，能让方案质量快速收敛到可用标准。

### 7.2 面向产品研发：能力优化方向
结合本次实测暴露的问题，从工程化层面给出可落地的产品优化建议，从根源上提升 Agent 输出的可信度：
1.  **执行层：强制关键步骤落地，禁止估算替代真实执行**
    涉及计数统计、版本核查、合规扫描等可通过命令完成的任务，应强制 Agent 调用系统工具实际执行，并返回执行结果原文，不允许仅凭文件片段、知识库信息输出确定性结论。
2.  **输出层：明确标注信息状态，区分“已执行”与“建议项”**
    审计、验证类输出必须增加状态标识，明确区分「已执行并验证」「引用自外部文档」「估算参考值」「建议执行步骤」四类信息，禁止以执行报告的格式呈现未实际运行的内容。
3.  **约束层：新增规则校验引擎，自定义规范强制遍历**
    对于用户提供的项目规范、铁律、禁则，应在方案生成后增加强制合规校验环节，逐条遍历核对并输出合规结论，不允许抽样检查后自称完成全量合规校验。
4.  **数据层：所有统计数据强制附带溯源信息**
    所有量化结论必须标注数据来源（对应执行命令、文件路径、行号），支持用户反向核验，从机制上消除“估算填数”的空间。

### 7.3 能力适用边界
结合本次实践表现，明确当前阶段 AI Agent 在架构级任务中的能力边界：
- **推荐独立完成的场景**：方案框架搭建、基础技术选型参考、标准化检查项枚举、已知问题的修正执行
- **需人工强兜底的场景**：
  - 要求行级精确统计的代码审计任务
  - 涉及跨模块、跨系统的依赖链推理与影响面分析
  - 存在大量自定义专属规则的合规性审查
  - 强业务语义关联的命名、重构类规划
  - 无人工复核的直接上线执行

***

### 八、结语
本次真实项目的全流程实测，呈现了与“AI 全自动提效”主流叙事不同的真实面貌：**在具备自定义规范、多模块依赖、业务专属语义的复杂架构任务中，当前阶段的 AI Agent 尚不能作为独立解决方案使用。**

它的价值不可否认：能在极短时间内完成方案框架搭建、基础问题枚举、标准化流程输出，将人类工程师从重复的基础梳理工作中解放出来，快速建立对项目的全局认知。
但它的短板同样显著：执行纪律不足、复杂因果推理有短板、自定义规则落地能力弱，导致初稿质量存在明显短板，人工纠偏、交叉验证的成本较高。在本项目的复杂度（97个文件、9条铁律、51条禁则）下，“AI 产出 + 人工深度校验”的模式，提效效果会受人工审查深度的直接影响。

更客观的定位是：**当前 AI Agent 是高效的辅助工具，而非全自动的替代方案**。它最终能发挥多大价值，既取决于产品本身的执行纪律与能力边界，也取决于使用者是否能建立合理的预期、用对人机协作的模式。
希望本次实测的完整复盘，能为产品迭代、社区实践提供一份可参考的真实样本。
