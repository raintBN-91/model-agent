# 基线协作策略

## 适用场景

当模型官网、官方 README、官方 benchmark 或官方文档没有给出可直接使用的训练或推理基线时，使用本策略。

## 强制规则

- 不要假设 GPU 基线天然存在。
- 不要在没有询问用户的情况下，直接把某组历史数据当作 GPU 基线。
- 不要因为本地 PyTorch 或 CPU 路径容易跑通，就把 CPU 输出当成最终精度基线。
- 不要在没有询问用户的情况下，默认由用户自行对比，或默认由 agent 直接读取某个目录。
- 必须先询问用户 GPU 基线准备方式，再继续。

## CPU 对比的合法用途

只有以下情况可以做 CPU 对比：

- 用户明确要求做 CPU vs NPU 检查
- 官方文档明确把 CPU 结果作为标准基线
- 为了定位 PyTorch 路径、算子行为或 dtype 漂移，临时做本地调试

如果属于第 3 类，必须在最终报告中明确写明“CPU 对比仅为调试辅助，不作为最终精度对齐结论”。

## CPU 对比的禁止用途

以下情况禁止把 CPU 结果拿来当正式基线：

- 因为 GPU 环境暂时不可用
- 因为本地 PyTorch 模型已经能在 CPU 跑通
- 因为 agent 已经写好了 CPU 推理脚本
- 因为想先快速给出一个 PASS 或 WARN 结论

这些都不能替代官方基线或用户确认的 GPU 基线。

## 可选协作方式

用户必须在明确确认后进入以下两种方式之一：

### 方式一：用户提供本地 GPU 基线目录

适用场景：

- 用户已经在 GPU 环境跑过训练或推理
- 用户能把结果文件同步到当前机器或工作区
- 用户希望 agent 继续读取文件并完成对比

需要用户提供：

- 本地目录路径
- 目录中包含的数据类型
- 每个关键文件的含义
- 哪一组文件是 GPU 基线，哪一组文件是昇腾结果

### 方式二：用户自行在 GPU 上运行并自行对比

适用场景：

- GPU 环境不在当前机器
- 用户不方便传输结果文件
- 用户只希望 agent 给方法，不希望 agent 直接读取数据

此时 agent 的职责是：

- 给出训练或推理对比步骤
- 给出必须核对的配置项
- 告诉用户需要保存哪些日志、指标和输出
- 告诉用户如何判断是否需要进一步做 dump

## 建议询问模板

在官方基线缺失时，至少问清以下内容：

- 你这边是否已经有 GPU 侧运行结果？
- 如果有，是否可以把训练日志、推理输出或指标文件放到一个本地目录，并把目录路径发给我？
- 如果没有，是否由你先在 GPU 上跑出结果后再继续对比？
- 你希望我直接读取本地目录继续分析，还是只给你对比步骤由你自行执行？

## 报告措辞要求

如果最终没有拿到官方基线或用户 GPU 基线，只做了 CPU 调试比较，报告必须写成：

- “当前只完成了 CPU vs NPU 调试性比较”
- “该结果不能替代官方基线或 GPU 基线”
- “不能据此宣称模型已经完成精度对齐”

## 升级到 Dump 前的前提

如果正式基线都尚未确认，不要直接进入 dump 排查。先把基线来源、数据位置和比对方式确认清楚。
